PytorchToBoard 当前项目规格页

本页只描述当前仓库里真实存在且仍在使用的部署链路。主路径是 NNFramework-main\run_model_to_board.py 驱动的零参数部署流程， 它会恢复固定输入目录中的 PyTorch 音频分类模型，导出多精度 TFLite 候选， 完成能力分析与验证，然后生成 nRF5340 固件需要的板端推理代码。

目标板 nRF5340
主入口 NNFramework-main\run_model_to_board.py
主运行时 TFLM + CMSIS-NN + CMSIS-DSP
工作区 D:\NRF5340\PytorchToBoard
当前仓库的核心用途是“模型部署到板端”和“nRF5340 固件集成验证”。

2. 当前结构

NNFramework-main

当前主部署流水线，负责输入检查、模型恢复、TFLite 导出、候选分析、验证与板端文件生成。

真实主流程从这里启动。

pencilv_stage_tflite_end2end

nRF5340 Zephyr 固件工程，负责音频采集、调用生成的推理运行时代码并输出分类结果。

真正参与编译和烧录的是这个固件工程。

3. 固定输入

当前主流程固定从 NNFramework-main\in\ 读取输入，不再要求用户通过命令行逐项传参。

路径 作用 说明
model.py PyTorch 模型定义 恢复阶段会按该文件中的结构重建模型。
train_1s.yaml 音频、特征、标签与数据路径配置 当前真实使用的配置文件名是 train_1s.yaml，不是旧文档中的 config.yaml。
唯一一个 .pt 文件 checkpoint 或 TorchScript 优先接受 best_model.pt 或 model.pt；如果存在多个 .pt 文件会直接报错。
data/ 真实音频数据目录 用于训练配置重放、代表性校准样本生成和真实任务验证。
calibration_data/ 辅助校准数据目录 作为输入布局的一部分保留在固定目录中。

4. 主入口流程

NNFramework-main\in\
  -> run_model_to_board.py
  -> 校验固定输入目录
  -> 重建 artifacts\latest_run\
  -> 写出 resolved_config.yaml
  -> python -m pipeline.cli run
  -> 选择最终 TFLite 候选
  -> pipeline\board\tflite2cppfile.py
  -> pencilv_stage_tflite_end2end\src\generated\
  -> 用户自行执行 west build / west flash

输入校验

主入口会检查 model.py、train_1s.yaml、唯一 .pt 文件和 data/ 是否齐全。

运行产物

所有本次部署的中间结果和日志会集中写入 NNFramework-main\artifacts\latest_run\。

板端输出

最终生成的 C/C++ 文件会写入 pencilv_stage_tflite_end2end\src\generated\，再由固件工程编译。

5. Pipeline 阶段

当前命令行入口 python -m pipeline.cli 暴露三个子命令：identify、recover 和 run。其中 run 是完整主流程。

阶段 职责 当前说明
identify 识别 checkpoint 类型和模型身份 生成输入清单与检查点静态识别结果。
restore 恢复 PyTorch 模型 把固定输入目录中的模型定义与权重恢复成可执行参考模型。
recovery 重建 process contract 恢复输入域、前处理步骤、模型输入输出语义与板端需要的约束元数据。
export 导出多精度 TFLite 候选 同时尝试 int8、float16、float32 三组候选。
analyze 分析可部署性和加速覆盖 检查 preprocess 步骤、算子支持情况以及当前板端运行时契约是否匹配。
validate 验证候选可用性 当前由 synthetic smoke 和 real-feature task validation 两层组成。
select 选择最终候选 选择顺序固定为 int8 > float16 > float32，必须同时满足导出成功、可部署、验证通过。
package 写出清单、报告与板端配置 统一生成 manifest、候选比较、验证报告与板端打包所需元数据。
选中的候选最终会折叠为三种部署状态之一：success、partial_success 或 failed。 如果发生伪校准回退、只有 synthetic smoke、并非完全加速或板端契约存在残余风险，结果会被标记为 partial_success。

6. 校准与验证

代表性校准优先

当前 int8 导出优先尝试基于真实音频窗口和 log-mel 特征重建代表性校准样本， 样本来自 train_1s.yaml 指向的数据目录，并按类别做平衡抽样。

允许伪校准回退

如果代表性校准样本无法构造，流程仍可能回退到 pseudo calibration。 这种情况下候选可以继续被选择，但部署状态会附带临时性风险告警。

验证分两层

每个候选都会先做 zeros、normal_seed_7、 uniform_seed_17 三组 synthetic smoke；若真实数据可用， 还会做基于真实特征的任务指标验证。

验证层 用途 当前行为
synthetic smoke 确认候选能稳定跑通 始终执行，用于检查量化输入准备、推理执行和输出有限性。
real-feature task validation 确认真实任务指标可接受 优先选择 test -> val -> train 数据切分，比较准确率、宏平均 F1 和参考模型 top1 一致率。

7. 板端运行时契约

当前板端生成器会用 process_contract.json 约束生成 runner 的配置，避免恢复结果、TFLite 模型与固件侧参数静默漂移。

输入域

只支持从原始音频 PCM 开始的音频域流程，不打包非音频输入域。

支持的前处理

当前生成 runner 显式覆盖 stft、mel、db_compression 和 normalize。

bandpass 现状

bandpass 目前只保留为元数据，不会在当前板端运行时里额外执行时域带通滤波。

normalize 限制

只接受 per_sample_standardize 或 disabled 两种模式。

当前生成 runner 要求 stft.center = false，并且会先按 canonical NCHW 重建前端特征，再适配到模型实际输入布局。 支持的布局标签包括 NCHW、NHWC、MATRIX 和 FLAT。 
当前可接受的特征形状语义示例
[1, 1, mel_bins, frame_count]
[1, mel_bins, frame_count, 1]
[1, mel_bins, frame_count]
[1, mel_bins * frame_count]

8. 输出产物

路径 内容
NNFramework-main\artifacts\latest_run\resolved_config.yaml 主入口根据固定输入目录重写后的本次部署配置。
NNFramework-main\artifacts\latest_run\pipeline\manifest.json 完整 pipeline 运行结果与最终部署状态汇总。
NNFramework-main\artifacts\latest_run\pipeline\candidate_comparison.json 多精度候选比较和最终选择结果。
NNFramework-main\artifacts\latest_run\pipeline\validation_report.json synthetic smoke 与真实任务验证报告。
NNFramework-main\artifacts\latest_run\pipeline\process_contract.json 恢复出的输入到输出处理契约。
NNFramework-main\artifacts\latest_run\pipeline\acceleration_report.json 各候选的加速覆盖与板端适配分析。
NNFramework-main\artifacts\latest_run\pipeline\unsupported_report.json 不支持项及其原因说明。
NNFramework-main\artifacts\latest_run\pipeline_run.log 主 pipeline 子进程标准输出和标准错误日志。
NNFramework-main\artifacts\latest_run\board_export.log 板端代码生成子进程日志。
pencilv_stage_tflite_end2end\src\generated\audio_event_model_*.h/.cpp 固件工程直接编译使用的生成代码，包括模型数据、配置和 runner。

9. 常用命令

执行模型到板端代码的主流程

python .\NNFramework-main\run_model_to_board.py

重建校准数据

python .\NNFramework-main\in\prepare_calibration_data.py

直接调用底层 pipeline

python -m pipeline.cli run ^
  --model-py D:\path\to\model.py ^
  --checkpoint D:\path\to\model.pt ^
  --config D:\path\to\train_1s.yaml ^
  --output-dir D:\path\to\out

编译 nRF5340 固件

cd .\pencilv_stage_tflite_end2end
west build -p always -b pencilv/nrf5340/cpuapp -s . -o=-j1 -- -DBOARD_ROOT=.

烧录固件

west flash

10. 环境假设

Python 侧

当前流程默认依赖 torch、tensorflow、onnx2tf、numpy 和 PyYAML。

固件侧

当前固件工程基于 Zephyr / west 构建，板端运行时使用 TFLM，并结合 CMSIS-NN 与 CMSIS-DSP。

当前边界

本页不再引用已经不在当前主路径中的旧训练脚本入口、旧测试套件目录或已经删除的补充 Markdown 说明。
